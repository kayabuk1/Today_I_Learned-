■ 現場監督のスローモーション（ログの「閾値」のアーキテクチャ） Pythonの logging モジュールには、ログの深刻度（プライオリティ）に応じて**5つの階級（レベル）**が明確に定義されています
。
CRITICAL（致命的・システムダウン）
ERROR（エラー・一部機能の停止）
WARNING（警告・まだ動くが注意） ⬅ ★システムのデフォルトの壁（閾値）
INFO（一般情報・正常な動作の記録）
DEBUG（デバッグ用・開発者向けの超細かいメモ）
現場監督：「よし、ログ出力の命令が次々と来ているな！ しかし俺の初期設定（デフォルト）の閾値は 『WARNING（警告）以上』 だ！」
現場監督：「warning、critical、error の3つは、警告以上の深刻な事態だから画面（標準エラー）に即座に出力しろ！」
現場監督：「しかし、debug と info は、俺の閾値（WARNING）に達していない（重要度が低い）から、握り潰して（抑制して）スキップしろ！」
これが、info と debug が画面に表示されなかった理由です。 実務において「すべての細かいアクセスログ（INFO）を表示させたい」という場合は、事前に現場監督の設定を logging.basicConfig(level=logging.INFO) のように明示的に引き下げる必要があります。
第2部：【真理】 なぜ print ではなく logging を使うのか？
インフラの自動化スクリプトを書く際、「画面に文字を出すだけなら print 関数でいいのでは？」と思うかもしれません。しかし、本番環境のサーバーで動かすコードにおいて print は御法度であり、必ずこの logging モジュールを使わなければならないという掟があります。
① 出力先の自在な切り替えができるから print は「今開いている画面」にしか文字を出せません。しかし logging を使えば、現場監督に「エラー（ERROR）が出たら画面に出しつつ、同時にファイル（app.log）にも書き込んで、さらにシステム管理者にメール（email）を飛ばせ！」といった高度なルーティング（送信先の選択）を一瞬で設定できます
。
② ログレベル（閾値）で情報量をコントロールできるから 普段（運用時）は閾値を「WARNING」にしておいて重大なエラーだけを監視し、システムに謎の障害が起きて調査したい時だけ、設定ファイルの閾値を「DEBUG」に下げる。すると、コードを1行も書き換えることなく、突然詳細なデバッグ情報が滝のように出力され始めます。
