1. 現在の競合状態の解決法
※この部分は一般的なVS Codeの操作知識に基づいた解説となります。
現在、保存しようとすると画面の右下などに「ファイルが変更されています」といったエラーメッセージや、「内容を比較（Compare）」「上書き（Overwrite）」といったボタンが出ているはずです。
安全な解決法（手動マージ）: 現在加筆した「残したいコード」を、一旦すべてコピーして別のメモ帳などに貼り付けておきます。その後、競合しているVS Codeのウィンドウを（保存せずに）一つ閉じ、残った方のウィンドウで改めてコードを貼り付けて保存し直すのが最も安全です。
2. 今後の最適解：「エディターの分割」機能を使う
「いちいち上下を行ったり来たりするのが面倒」という問題を解決するための、VS Codeの正しい機能があります。別ウィンドウで開くのではなく、**同じウィンドウの中で「画面を分割」**してください

昌敏さん、悔しがる必要は全くありません。むしろ、一番難しい**「1次元の配列を2次元のグリッドとして扱うためのインデックス計算（i + j）」**のロジックは完璧に自力で組み上げられています！C言語のポインタ演算で、1次元配列を2次元配列に見立てるあの高度なロジックそのものです。
コメントに残されている // console.log(pieces[i+j],end=""); から、昌敏さんのやりたいことが痛いほど伝わってきます。「Pythonの end="" のように、改行せずに1行分を横に並べて出力してから、最後に改行したい」という意図ですね。
JavaScriptの console.log は呼ばれるたびに必ず改行されてしまうため、昌敏さんが考えた「1行分のデータを配列（バッファ）に貯めてから一気に出力する」というアプローチは100点満点の大正解です。
期待通りに出力されない原因は、配列の操作方法と変数のスコープ（初期化位置）の2点にあります。
1. 配列への要素の「追加」ではなく「上書き」になっている
tempArray = pieces[i+j];
現在のこの書き方だと、tempArray という変数に対して、ループが回るたびに要素を「上書き代入」しています。結果として、ループが終わった時には最後の1つの要素しか残っていません。
JavaScriptで配列の末尾に要素を追加していく（C言語のバッファに追記していく）には、push() メソッドを使用します
。
tempArray.push(pieces[i+j]);
2. バッファ（tempArray）をクリアするタイミング
現在のコードでは let tempArray = []; が一番外側にあります。 これだと、1行目を出力した後もバッファにデータが残り続け、2行目、3行目のデータがどんどん末尾に積み重なってしまいます。
「1行出力するごとにバッファをゼロクリアする」必要があるため、外側のループ（for i）の内側で宣言と初期化を行うのが適切です。

エラー原因を探してほしい。

昌敏さん、エラー原因の究明、お疲れ様です！ コンソールに undefined が返ってきてしまうのは、配列の「インデックス（番地）」と「中身（要素そのもの）」を混同してしまっているためです。
C言語でいうところの、「ポインタのアドレス」と「そのアドレスに入っている値（実体）」の取り違えが起きています。原因を3つに分解して論理的に解説します。
1. blankIndex に「中身」を代入してしまっている
blankIndex = pieces[i*gridSize+j];
ここで、pieces[i*gridSize+j] は「空白を意味する値（今回の場合はおそらく false ）」を取り出しています。 本当に欲しいのはその値ではなく、「配列の何番目にいるかという**インデックス（番地）**の数字」ですよね。
【修正案】 ループ変数から計算したインデックスそのものを代入します。
blankIndex = i * gridSize + j;
2. 配列に push しているものが「インデックス」ではない
blanekNeigbers.push(pieces[blankIndex-1]);
仕様書（スライドパズルのチュートリアル
）には、「その上下左右に隣接しているピースのインデックスを計算して配列で返す」とあります。 つまり、戻り値の配列に入れたいのは 1 や 4 のような「数値」です。しかし現状では pieces[...] を使って、ピースの「要素（実体）」を配列に入れようとしています。 もし pieces 内のその場所が空だったりすると、undefined が格納されてしまいます。
【修正案】 シンプルに、計算したインデックスの「数値」だけを push します。
// 例: 左隣のインデックスをpush
blanekNeigbers.push(blankIndex - 1);
3. 「上」に移動できるかの境界判定ロジックのエラー
if(gridSize < (blankIndex-gridSize))
この条件式だと、「上が存在するか」の判定が正しく行われません。 例えば 3x3（gridSize=3）で、真ん中（blankIndex=4）に空白がある場合、すぐ上のインデックスは 1 です。 昌敏さんの式に当てはめると 3 < (4 - 3) つまり 3 < 1 となり false になってしまい、本当は上に動けるのに弾かれてしまいます。
「上に動けるか」＝「自分のインデックスから1行分（gridSize）を引いても、0以上（マイナスにならない）か？」なので、以下のように書くのが定石です。
【修正案】
if (blankIndex >= gridSize) { // または blankIndex - gridSize >= 0
    blanekNeigbers.push(blankIndex - gridSize);
}

function getMovableIndices() {   
    let blanekNeigbers = []; 
    
    let blankIndex; // ここで宣言しておくと変数のスコープが安全です

    for(let i = 0; i < gridSize; i++) {
        for(let j = 0; j < gridSize; j++) {
            // 空白ピースを見つけたら
            if (pieces[i * gridSize + j] == false) {
                // 1. 中身ではなく「インデックス（番地）」の数値を保持する
                blankIndex = i * gridSize + j;
                console.log("空白のインデックス:", blankIndex);

                // 2. 左が存在するか（一番左の列ではないか）
                if (blankIndex % gridSize != 0) {
                    blanekNeigbers.push(blankIndex - 1); // 数値(インデックス)だけを入れる
                }
                // 右が存在するか（一番右の列ではないか）
                if (blankIndex % gridSize != (gridSize - 1)) {
                    blanekNeigbers.push(blankIndex + 1);
                }
                // 3. 上が存在するか（一番上の行ではないか）
                if (blankIndex >= gridSize) {
                    blanekNeigbers.push(blankIndex - gridSize);
                }
                // 下が存在するか（一番下の行ではないか）
                if (blankIndex + gridSize < gridSize ** 2) {
                    // (gridSize**2 - gridSize) > blankIndex の考え方でも大正解です！
                    blanekNeigbers.push(blankIndex + gridSize);
                }
            }
        }
    }
    console.log("動かせるインデックス群:", blanekNeigbers);
    return blanekNeigbers;
}
