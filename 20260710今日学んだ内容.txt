【コマンド連続実行の3つの文法規則】
1. 無条件の連続実行： ;（セミコロン）
文法：コマンド1 ; コマンド2
意味：左のコマンド1の実行が成功しようが失敗しようが（エラーになろうが）関係なく、終わったら無条件で右のコマンド2を順々に実行します
。
今回のケース：whoami を実行し、それが終わったらすぐに hostname を実行するという最もシンプルな連続処理です
。
2. 成功した時だけ次を実行： &&（ダブルアンパサンド）
文法：コマンド1 && コマンド2
意味：左のコマンド1が成功（システム内部の終了ステータスが0）した場合のみ、右のコマンド2を実行します
。
インフラ的活用（防弾装甲）：プロの現場では非常に多用されます。例えば cd /share && rm *.txt と書けば、「shareフォルダへの移動に成功した時だけ、テキストを全削除する」という安全な処理になります（もし移動に失敗したら削除処理はキャンセルされます）。
3. 失敗した時だけ次を実行： ||（ダブルパイプ）
文法：コマンド1 || コマンド2
意味：左のコマンド1が失敗（システム内部の終了ステータスが0以外）した場合のみ、右のコマンド2を実行します
。
インフラ的活用：コマンド1がエラーになった時の「例外処理（バックアップやエラー通知）」として使われます。「ファイルが見つからなければ、エラーメッセージを出す」といった具合です。
■ 統合結論
今回昌敏さんが打ち込んだ ; による連続実行は、ターミナル上で一回一回エンターキーを押して待つ手間を省き、「1行の強力なプログラム（ジョブ）」としてOSにまとめて命令を下すためのスマートなハッキング手法です

1. 【モジュール解剖】：urllib とは何か？
urllib（ユーアールエル・リブ）は、URL（インターネット上の住所）を扱うためにPythonが標準で用意している巨大なライブラリ（部品箱）です。
その中にはいくつかの専門部署がありますが、今回呼び出した urllib.parse は、「URLの解析（パース）と結合」に特化した専門部署です。
2. 【機能解剖】：urljoin(ベースURL, リンク) の強力な配管マジック
ab_url = urllib.parse.urljoin(url, link) この1行は、単なる「文字列の足し算（url + link）」ではありません。システム内部で非常に賢い**「文脈の判断と結合」**を行っています。
第1引数（url）：現在のページ（基準となるベースURL）。例：https://www.pref.chiba.lg.jp/kg-funabashi/index.html
第2引数（link）：ページ内から抽出したリンク先。
動作の真実：
もし link がすでに絶対URL（http... で始まる）だった場合、システムは「結合の必要なし！」と判断し、link のまま返します。
もし link が相対URL（/about.html や ../images/logo.png など）だった場合、ベースURLの階層構造（ディレクトリ）を正確に計算し、足りないドメイン名やパスを完璧に補って結合した絶対URLを返します。
3. 【真相解明】：なぜ正規表現（re）ではなく urljoin を使うべきなのか？
昌敏さんが「正規表現でやるのかと思った」と考えたのは、文字列操作の観点からは大正解です。しかし、実際のWebサイトにおける相対URLには、**正規表現だけでさばくには危険すぎる「エッジケース（例外）」**が大量に潜んでいます。
パターンA：/info.html （一番上の階層からスタート）
パターンB：./info.html （今の階層からスタート）
パターンC：../info.html （1つ上の階層に登ってからスタート）
パターンD：#section1 （同じページ内のジャンプ）
パターンE：javascript:void(0) （プログラムを動かすだけのリンク）
これらすべてを正規表現の re モジュールだけで分岐させようとすると、コードがカオス（スパゲッティ状態）になり、必ずバグが発生します。 urllib.parse.urljoin は、こうした**「URL特有の複雑な階層ルールや例外」をシステム側で安全に吸収し、一発で正しい絶対URLに変換してくれる『最強の防弾配管』**なのです！

1. 【真相解明】：pathlib.Path と os のアーキテクチャの違い
osモジュールかと思ったら違うモジュール。osモジュールとはどう違う？ Pathと大文字になっているので、クラス自体をインポートしている？
■ 結論：大正解です！Pathはクラスであり、osが「文字列の足し算」でパスを作る旧世代の配管なのに対し、pathlibはパスを「高機能なオブジェクト（箱）」として扱う新世代のモダンな配管です。
クラスの直接インポート： 昌敏さんの推測通り、from pathlib import Path は、「pathlib という巨大な部品箱の中から、Path という設計図（クラス）だけをピンポイントで引き抜いてこい」という命令です。Pythonのインフラルール（PEP8）において、大文字で始まる名前は「クラス」を意味します。
os と pathlib の設計思想の違い：
os モジュール：パスをただの「文字列」として扱います。例えばフォルダとファイル名を繋ぐ時、os.path.join(folder, filename) と関数に文字列を渡して処理させます
。
pathlib モジュール：パスそのものを「オブジェクト（機能を持った箱）」として実体化させます。out_folder = Path("download_image") とすることで、out_folder 自身が「私はディレクトリを作る機能（mkdir）を持っています！」と自己主張できるようになります。だからこそ out_folder.mkdir() と直感的に書けるのです。
mkdir と os.makedirs の違い： やっていること（ディレクトリの作成）は全く同じです
。ただ、関数に命令する（os）か、オブジェクト自身に実行させる（pathlib）かという、システム設計（アーキテクチャ）の違いに過ぎません。現在のPython開発では、より直感的でバグが起きにくい pathlib を使うのがプロの定石となっています！

【完成版：課題「Soup_kadai.py」提出用コード】
import requests
from bs4 import BeautifulSoup
import datetime
import re
import urllib
import time
from pathlib import Path

find_target1 = "img"
find_target2 = "src"

# ターゲット（例としてWikipediaのRaspberry Piページ）
url = "https://ja.wikipedia.org/wiki/Raspberry_Pi"

headers = {"User-Agent": "Mozilla/5.0"}
response = requests.get(url, headers=headers)

print(f"通信ステータス: {response.status_code}")
response.encoding = response.apparent_encoding
soup = BeautifulSoup(response.content, 'lxml')

# 課題指定のディレクトリ作成
out_folder = Path("download_all")
out_folder.mkdir(exist_ok=True)

# //の直後から、次の / までの文字を取り出す
match = re.search(r'//([^/]+)', url)
if match:
    domain = match.group(1)
    print(f"抽出ドメイン: {domain}")

# 【追加】画像取得の制限回数を設定するカウンター
count = 0
limit_count = 3 # 3回（3枚）でストップ

print(f"画像収集ミッション開始（上限: {limit_count}枚）...")

for element in soup.find_all(find_target1):
    # 制限回数に達したらループを破壊して脱出する（break）
    if count >= limit_count:
        print("制限回数に到達したため、通信を安全に遮断します。")
        break
        
    src = element.get(find_target2)
    
    # 【防弾装甲】srcが空でない場合のみ実行
    if src:
        image_url = urllib.parse.urljoin(url, src)
        img_filename = image_url.split("/")[-1]
        
        # 画像を取得（画像取得時も身分証 headers を提示すると安全です）
        imgdata = requests.get(image_url, headers=headers)
        
        out_path = out_folder.joinpath(img_filename)
        with open(out_path, mode="wb") as file:
            file.write(imgdata.content)
            
        print(f"[{count+1}/{limit_count}] 保存完了 >> {img_filename}")
        
        # カウンターを1増やす
        count += 1
        
        # 【超重要】サーバーへの配慮（DoS攻撃回避）のため、ループ内で1秒待機
        time.sleep(1.0)

print("全ミッション完了！")
