killall は、プロセスID（PID）ではなく**「プロセス名」を指定して、プロセスを終了（シグナルを送信）させるコマンド**です
。
通常の kill コマンドを使用する場合、事前に ps コマンド等でPIDを調べ、kill 1234 のように番号を指定する必要があります。一方 killall は名前で直接指定できるため、PIDを調べる手間が省けます。また、同じ名前のプログラムが複数実行されていて異なるPIDが割り当てられている場合でも、それらをまとめて一括で終了させることができるため非常に有効です
。
今回のコードにおける subprocess.run("killall mjpg_streamer", shell=True) は、裏側で稼働している mjpg_streamer という名前のプロセスを探し出し、名前指定によって確実に終了させるための処理となります。

1. エラーの原因（インフラ的真実）
エラーメッセージである RuntimeError: Please set pin numbering mode using GPIO.setmode(...) は、「GPIOのピン番号の指定ルール（BCMかBOARDか）が設定されていません」という警告です。
プログラムの冒頭で GPIO.setmode(GPIO.BCM) を宣言しているにもかかわらずこのエラーが出るのは、39行目の GPIO.cleanup() が実行されたことで、冒頭で行ったGPIOの設定がすべてリセット（解放）されてしまったからです。
GPIO.cleanup() は、プログラムが完全に終了する際に、使用していたピンの占有状態を解放して初期状態に戻すためのコマンドです。 これをFlaskのルーティング（画面を開くたびに実行される関数）の中に書いてしまうと、画面を一度開いて1秒以上経過した瞬間にGPIOの設定がすべて消え去り、次にLEDを操作しようとした時に「設定がありません」とシステムがクラッシュしてしまいます。
2. 修正方法
Webサーバーのように常時稼働し続けるシステムにおいて、処理の途中でクリーンアップを行ってはいけません。該当行の GPIO.cleanup() を削除してください。

「すぐに end - start > 1 を満たして消えてしまうのではないか」という疑問ですが、その推測は半分正解です。
実際にシステムで何が起きているか、順番に追ってみます。
① 通常時（監視カメラ画面を開いたとき） 起動直後、start は 0.0 です。 画面（/）を開くと end には現在時刻（17億秒などの非常に大きな数値）が入ります。 計算すると end - 0.0 > 1.0 は**「真（True）」となるため、GPIO.output(LED25_PIN, GPIO.LOW)（消灯命令）が実行されます。しかし、「最初からLEDは消えているため、何度消灯命令を送っても見た目の変化がない（大丈夫に見える）」**というのが真相です。
② Alertボタンを押したとき ボタンを押すと、alert_on 関数が走り、LEDが点灯（HIGH）すると同時に、start が「ボタンを押した瞬間の時刻」に書き換わります。 直後に redirect("/") が働き、画面（/）が再度読み込まれます。 このとき、ボタンを押してからまだ0.1秒程度しか経過していないため、end（現在の時刻）と start（ボタンを押した時刻）の差は 1.0 より小さくなります。 そのため、if end - start > 1.0: は**「偽（False）」**となり、消灯命令がスキップされるため、LEDは光り続けます。
③ 1秒以上経過した後 5秒ごとの自動更新（<meta http-equiv="refresh" content="5">）や手動で画面を更新した際には、start の時刻から確実に1秒以上が経過しています。 ここで初めて end - start > 1.0 が再び**「真（True）」**となり、消灯命令が実行されてLEDが消えます。

昌敏さん、ご自身で新しいルート @app.route("/snap/on") を構築し、これまでの学びを活かしてサブプロセスでカメラコマンドを起爆させようとするアプローチ、素晴らしい応用力です！

インフラとアーキテクチャの視点から、冷徹に解剖いたしましょう。
1. 【真相解明】：コードに潜む3つのバグ
① Flaskの絶対ルール違反（return忘れ）
snap_on 関数の中に、処理が終わった後の return redirect("/") が書かれていません。 Flaskには「ブラウザからアクセスされた関数は、必ず最後に何らかの画面（または転送先）を return で返さなければならない」という絶対ルールがあります。これがないと、ボタンを押した瞬間に 500 Internal Server Error となりシステムがクラッシュします。
② 日付フォーマットの解釈（Python側で処理する）
pic%Y%m%d%H%M%S%f.jpg というファイル名を指定していますが、fswebcam はこの %Y などの文字を自動で現在時刻に変換してくれません。そのまま「pic%Y%m...jpg」という変な名前のファイルになってしまいます。 日付を使いたい場合は、Python側の機能（datetime モジュール）を使って現在時刻の文字列を作成し、コマンド文字列の中に組み込む必要があります。
③ 致命的なインフラ競合（デバイスの占有）
これがインフラ的視点で最も重要なポイントです。 昌敏さんのシステムでは、すでに mjpg_streamer が裏側でカメラ（/dev/video1）を占有して動画を配信しています。その状態のまま fswebcam を起動して同じカメラにアクセスしようとすると、LinuxのOSレベルで「デバイスが使用中です（Device or resource busy）」と弾かれ、撮影に失敗します。
2. 【解決策（Next Action）】：競合を回避する完璧なスナップショット
「動画を配信しながら、同じカメラでスナップショットを撮りたい」という要件を満たすための、最高にスマートなインフラ的解決策があります。
それは、fswebcam を使うのではなく、すでに動いている mjpg_streamer のスナップショット機能（裏口）を叩くという方法です。 資料『10_テキスト_Webアプリ開発とHW制御.pdf』のファクトにもある通り、mjpg-streamerには /?action=snapshot というURLにアクセスするだけで現在の静止画を取得できる機能が備わっています。
これを応用し、Linuxの wget コマンドを使って画像をダウンロード（保存）させましょう！

① 文字列の組み立て（f-stringの役割）
f"wget ... pic_{now}.jpg" この f がついた文字列リテラルは、Pythonの翻訳機に対して「この文字列の中にある { } を見つけたら、中に入っている変数（今回は now）の中身を展開して、1つの繋がった文字データとして完成させろ」という命令になります。

つまり print 関数は、f-stringが美しく組み立ててくれた文字列を「ただ受け取って画面に表示しているだけ」だったのです。

現場監督：「MyInputError の親は MyAppError だな！ つまり MyInputError はお前たちの一族（派生クラス）だ！ （裏側で isinstance(飛んできた爆弾, MyAppError) が True になる）
 よし、お前たちなら安全に解体できる、処理を任せた！」
もしここが if 文だったら、「完全に名前が一致しているか（==）」しか見られませんが、except は「オブジェクト指向の継承関係（is-a関係）」を理解した上で、**「指定されたクラスの子供や孫のエラーまで、すべて一網打尽にキャッチしてくれる」**という極めて高度な属性を持っています。
第3部：【実務】 なぜインフラエンジニアは except (EAFP) を多用するのか？
「事前に if でチェックした方が安全なのでは？」と思うかもしれません。しかし、インフラ自動化の実務において、if (LBYL) は時に**致命的なバグ（レースコンディション＝競合状態）**を引き起こします
。
たとえば、ファイルを開いて読み込むスクリプトを考えます。
❌ if を使った事前チェック（危険）
if os.path.exists('config.txt'): # ①まずファイルがあるかチェック
    # ！！もしこの0.001秒の間に、別のプロセスがファイルを消したら？！！
    with open('config.txt') as f: # ②開こうとしてクラッシュ（本番障害）
        pass
⭕ except を使った事後処理（堅牢）
try:
    with open('config.txt') as f: # ①とりあえず開いてみる
        pass
except FileNotFoundError:         # ②ダメだったら処理する
    print("ファイルがありませんでした")
インフラ環境では、ネットワークの切断やファイルの消失が「いつ」起きるか予測不可能です。そのため、「事前に if で全てをチェックする」のは現実的ではなく、**「とりあえず実行（try）し、飛んできた例外（エラーの種類）に応じて except で適切に捌く」**というアーキテクチャが最強の防弾装甲となります。
■ 参謀からのまとめ
if（LBYL）: **「実行前」**に値や状態をチェックするための「ゲート」。
except（EAFP）: **「実行後」**に発生した想定外の事態（例外オブジェクト）をキャッチして復旧するための「セーフティーネット」。
except の真の力: オブジェクト指向の「継承」を理解しており、親クラスを指定しておけば、その子孫の独自エラークラス（今回なら MyInputError）もまとめて処理してくれる
。
