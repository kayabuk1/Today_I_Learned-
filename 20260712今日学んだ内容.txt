255

1. 実際にどうなるのか？（手口の正体）
この手口は**「中間者攻撃（Man-in-the-Middle Attack）」**と呼ばれます。
罠の設置: 攻撃者は、昌敏さん（クライアント）と本物のサーバの間のネットワーク（例えば、公衆の怪しいWi-Fiなど）に潜り込みます。
偽の身分証の提示: 昌敏さんが初めてサーバにアクセスした瞬間、攻撃者が横入りして**「攻撃者自身の公開鍵（偽の身分証）」**を昌敏さんに提示します。
罠の発動: 昌敏さんが「これが本物のサーバの鍵だな」と信じて「yes」と打ってしまうと、昌敏さんの known_hosts ファイルには攻撃者の公開鍵が登録されてしまいます。
筒抜けの通信: 以降、昌敏さんが送るパスワードや秘密のデータは、すべて攻撃者の鍵で暗号化されて送られます。攻撃者はそれを自分の秘密鍵でスッと解読（盗聴）し、何食わぬ顔で本物のサーバへ横流しします。
つまり、最初の入口を改ざんされて「yes」と認めてしまうと、それ以降の暗号化通信がすべて攻撃者に筒抜けになるという致命的な事態に陥ります。
2. どうやって防ぐのか？（フィンガープリントの登場）
「じゃあ最初の1回目はどうやって見分ければいいんだ！」となりますよね。 実は、昌敏さんがDebianに初めてSSH接続した時の**「セキュリティ警告」の画面**に、その防衛策がしっかり表示されていたのです。
資料の

「known hostsリストにサーバ 192.168.52.132 のエントリはありません、悪意を持ったホストが、接続しようとしているサーバのふりをしている可能性もありますので、十分注意してください!」
 「サーバ側のホスト鍵指紋: SHA256:6RMzapC9azhTObxQEcOKToxwS05iyUZFwbNAD3VWTaM」

この 「フィンガープリント（指紋）」 こそが、唯一の防衛線です。
フィンガープリントとは： 長すぎる公開鍵の文字列を、MD5やSHA256といった暗号計算で短く要約した「ハッシュ値（指紋）」です

正しい「初回の作法」： 実務の世界では、初めてサーバに接続する際、画面に出たこの「指紋の文字列」をすぐには「yes」と打ちません。 **「事前に、サーバの管理者から直接（電話や社内チャット、手渡しなど、絶対に安全なルートで）本物の指紋の文字列を教えてもらっておく」**のです。 そして、画面に出た指紋と、手元のメモの指紋を「目視で照合」し、1文字も違わず完全に一致した時だけ、初めて「yes」と打ちます。

2. 「auth~」から始まる英単語の解剖（認証 vs 認可）
ここ、ITエンジニア（特にセキュリティ分野）が必ず一度は混乱する「英語の壁」です！ 昌敏さんが思い出そうとしていた残り2つを含めて、**「IT業界のAuth四天王」**を論理的に整理します。
① Authentication（オーセンティケーション）＝「認証」
意味：「お前は誰だ？」「本当に本人か？」を証明・確認すること。
例：パスワードを入力する、公開鍵認証で秘密鍵を持っているかテストする。
② Authorization（オーソライゼーション）＝「認可」
意味：「本人なのは分かった。じゃあ、お前にアクセスを許可するぞ」と権限を与えること。
例：今回の正解である authorized_keys。これは authorized（認可された・許可された）＋ keys（鍵たち）で、「この鍵を持ってる奴は、このサーバーに入ってヨシ！」とアクセスを許可された鍵のリストという意味になります。
③ Authority（オーソリティ）＝「権威・権限」
意味：許可を与えたり、身元を保証したりする「お上（機関）」。
例：Certificate Authority（CA：認証局）。SSL証明書などを「このサイトは安全な本物だ！」とお墨付きを与える機関のことです。
④ Authenticity（オーセンティシティ）＝「真正性（本物であること）」
意味：偽物じゃなく、正真正銘本物であるという性質。
例：初めてSSH接続した時に The authenticity of host ... can''t be established.（ホストの真正性が確立できません）という警告が出ましたよね
。まさにあの時の単語です！

**Authority（総務部）**が発行した社員証を使って、
受付で「私は昌敏です」と証明するのが Authentication（認証）。
ゲートのシステムが社員証を読み取り、「この人はサーバールームに入ってヨシ！」とドアを開けるのが Authorization（認可）。
そもそもその社員証が偽造されていない本物であるという性質が Authenticity（真正性）。

ファイルに直接書き込む： # echo 1 > /proc/sys/net/ipv4/ip_forward

② 経路（ルーティング）の「教え方」が違う

route コマンドを使う場合： # route add -net 172.17.0.0 netmask 255.255.0.0 gw 172.16.255.254
ip route コマンドを使う場合： # ip route add 172.17.0.0/16 via 172.16.255.254
Ciscoの文法に似ていますが、Linuxの ip コマンドは「目的語（route）＋動詞（add）」のような独特の語順を持っています

2. なぜ complete ではなく completion なのか？（命名の法則性）
「動詞か名詞か、法則性があればいいのに」という昌敏さんの着眼点、10000%大正解です！ 実は、Linuxのパッケージやファイルの名前には、英語の品詞に基づいた非常に強力な「暗黙の法則」が存在します。
【大原則】 パッケージ名は「モノ（機能）」なので、必ず「名詞」または「名詞形」になる。
パッケージ（箱）は、システムに追加する「部品」や「機能という物体」です。そのため、「〜する（動詞）」ではなく、「〜すること/もの（名詞）」という名前が付けられます。
complete (動詞：補完する) 👉 completion (名詞：補完機能)
だから bash-completion（Bashの補完機能セット）になります。
configure (動詞：設定する) 👉 configuration / config (名詞：設定)

epel* （後ろだけ）: 「epel から始まる」ものを探す。
*epel* （両端）: 前に何文字あっても、後ろに何文字あっても、**「途中に epel が含まれている」**ものをすべて探す。

1. 「契約更新（source）」の魔法の正体
以前「設定ファイルを読み直す」とお伝えした source ~/.bashrc ですが、裏側では**「親（現在のシェル）と子（子シェル）の次元の違い」**という非常に重要なロジックが動いています。
Linuxでプログラム（スクリプト）を実行する際、通常は以下のような動きになります。
通常の実行（bash script.sh など） 大精霊（現在のBash）は、**「新しい子どもの精霊（子シェル）」**を生み出し、その子どもに作業をやらせます。子どもは作業が終わると消滅するため、子どもが設定した変数や環境は、親（今の画面）には一切引き継がれません
。
source コマンドでの実行 source （または .）を使うと、子どもを作らずに**「今の自分自身（現在のシェル）」**の中で設定ファイルを実行します
。そのため、エイリアスや履歴の設定などが、そのまま「今の画面」にバッチリ反映されるのです。

ここには、**「組み込みコマンド」と「外部コマンド」**という決定的な違いが隠されています。
外部コマンド（ls、cat、bash など）： これらは /usr/bin/ls のように、**実体のある「ファイル」**として存在します。
組み込みコマンド（source、cd、echo など）： これらは、**「執事（Bashシェル）が最初から持っている特殊能力（内蔵機能）」**であり、独立したファイルとしては存在しません。
【Dockerの勘違い（エラーの真相）】 docker run というコマンドは、「コンテナという家を作ったら、次に『実体のあるファイル』を直接起動しよう」とする性質があります。 つまり、Dockerは「source という名前の実行ファイル（道具）」を家の中で必死に探したのですが、source は執事（Bash）の能力であって物理的な道具ではないため、「そんなファイル無いぞ！」とパニックを起こしてしまったのです。

docker run: イメージからコンテナを生成して起動し、最初のコマンドを実行する

docker exec: すでに実行中のコンテナの中で、新しいプロセス（コマンド）を追加で実行する

-it（どうやって通信するか）: 「執事と対話するために、キーボード入力（-i）を繋ぎっぱなしにして、黒い画面の窓口（-t: 擬似端末）を用意しろ！」という指定です
