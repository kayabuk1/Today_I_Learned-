#%% 9.1.5 クロージャー　関数閉方
#ローカルスコープは関数の呼び出しの度に生成される。
# =c1とc2での変数countは別物
def counter(init):
    count = init
    def increment():
        nonlocal count
        count += 1
        return count
    return increment
c1 = counter(1)
#counter()で呼び出された時点で 1に＋1する関数incrementが定義され、
#戻り値としても戻された、increment関数オブジェクトがc1名札に紐づけられる。
#c1（）はcallableなので呼び出して実行ができる。
c2 = counter(25)
print(c1())
print(c1())
print(c2())
print(c2())

#Q.yieldとはどんな由来語源核となるイメージの単語？
#英単語の yield（イールド）には、大きく分けて以下の2つの意味
#そして、プログラミングにおける yield は、この両方のニュアンスを同時に体現しています。
#（農作物や利益などを）産出する、生み出す」：工場から製品（値）をポンッと外に出す。
#「（権利や道を）譲る、一時停止して明け渡す」
#作業を「一時停止（フリーズ）」して、呼び出し元の相手に処理の主導権を譲る。
#つまり yield の核となるイメージは、
#   **「値を一つだけ『産出』したら、そこで自分自身の動きをピタッと
#   『一時停止（主導権を譲る）』して、次の呼び出しが来るまで待機する」

第1部：【解剖】 パーサーの判断と @ の意味
Q. パーサーは @ を見るとどんな判断をする？ @ の意味は？
【結論】 パーサーにとって @ は、**「これから定義される関数を、指定した改造工場に入れて、元の名前のまま上書きしろ！」という絶対的な命令（シンタックスシュガー＝甘くコーティングされた省略記法）**です。公式ドキュメントでも、デコレータを導入する「デリミタ（区切り文字）および演算子」として定義されています
。
コードの下半分に書かれた以下の部分を見てください。
@deco
def test():
    print("hello")
この部分は、システムの裏側では昨日昌敏さんがトレースした以下のコードと全く同じことをやっています
。
# 1. 普通に test 関数を作る

    print("hello")

# 2. test 関数を deco 工場に放り込み、
#    出てきた inner 関数に「test」という名札を貼り直す！
test = deco(test)

第2部：【解剖】 deco の正体と後ろに続く関数
Q. deco は予約語？
【結論】 いいえ、予約語ではありません。 func や inner と同じく、人間が自由につけたただの「関数の名前（名札）」です。ここでは「デコレータ（decorator）」の役割を持たせているため、分かりやすく deco と名付けられているだけです。
Q. @ の後ろには deco 以外にも入る関数はあるの？
【結論】 はい、無限にあります！ 「関数を1つ受け取って、新しい関数を返す」ように設計された高階関数（改造工場）であれば、どんな関数でも @ の後ろに置くことができます。

第1部：【解剖】 yieldの語源と、核となるイメージ
Q. yieldとはどんな由来語源核となるイメージの単語？
【結論】 英単語の yield（イールド）には、大きく分けて以下の2つの意味があります。そして、プログラミングにおける yield は、この両方のニュアンスを同時に体現しています。
「（農作物や利益などを）産出する、生み出す」
イメージ：工場から製品（値）をポンッと外に出す。
「（権利や道を）譲る、一時停止して明け渡す」
イメージ：自分は作業を「一時停止（フリーズ）」して、呼び出し元の相手に処理の主導権を譲る。
つまり、Pythonにおける yield の核となるイメージは、**「値を一つだけ『産出』したら、そこで自分自身の動きをピタッと『一時停止（主導権を譲る）』して、次の呼び出しが来るまで待機する」**という状態です。

第2部：【真理】 return と yield の決定的な違い
これまでの関数では、結果を返すために return を使っていました。
return の工場：製品を一つ出荷したら、その瞬間に工場は閉鎖され、中の記憶（ローカル変数など）は全てシュレッダーにかけられます。
yield の工場（ジェネレーター）：製品を一つ出荷したら、シャッターを下ろして「一時停止」するだけです。工場内の機械の状態や変数の記憶は完全に保存されたまま維持されます
。そして、「次ちょうだい」と言われたら、前回一時停止した場所から何事もなかったかのように作業を再開します
。
Pythonは、関数の中に yield という文字を一つでも見つけると、パーサは「おっ、これは普通の関数ではなく『ジェネレーター関数』だな！」と認識を切り替えます
。

第1部：【解剖】 「式の途中で中断する」という大正解
昌敏さんの推測: 式の途中でも中断するのということか？
【結論】 10000%その通りです！！代入される直前で時が止まっています！
公式ドキュメント（6.16. 評価順序）には、**「代入式を評価するときは、右辺が左辺よりも先に評価されます」**と明確に定義されています

出荷と中断: 次に yield '横須賀' が実行されます。ここで工場は '横須賀' を外（for ループの name）に出荷し、左辺の n =  に代入する一歩手前で、ピタッと「一時停止（中断）」します。

1. 【真相解明】：apt と apt-get は何が違うのか？
まず大前提として、Linuxでソフトウェア（パッケージ）を管理するコマンドには歴史的な進化の過程があります。
apt-get や apt-cache（旧世代・専門職人） これまでLinuxのパッケージ管理は、**「インストールや削除を行う職人（apt-get）」と、「パッケージの検索や情報表示を行う職人（apt-cache）」**というように、機能ごとにコマンドが完全に分かれていました
。しかし、人間がこれらを使い分けるのは面倒でした。
apt（新世代・統合インターフェース） そこで近年登場したのが apt コマンドです。これは apt-get や apt-cache でよく使われる主要な機能を1つのコマンドに統合・洗練させたモダンなコマンドです
。 （※現在のラズパイ開発では sudo apt install [パッケージ名] のように、apt を使用するのが主流かつ推奨となっていますが、裏側の動作は同じです
）。

2. 【アーキテクチャ解剖】：主要な apt- 系コマンドの役割
昌敏さんが引き出したコマンドリストの中で、インフラエンジニアが主に使う「3つの職人（コマンド）」の役割を解剖します
。
① apt-get（配備・撤去職人） パッケージの追加、更新、削除など、システムに変更を加える実働部隊です
。
② apt-cache（検索・情報収集職人） システムを変更せず、データベースからパッケージの情報を検索・参照するための情報部隊です
。
search：キーワードでパッケージを検索する
。
show：パッケージの詳細や依存情報を表示する
。
③ apt-file（逆引き探索職人） 「このファイル（例：sshd）は、どのパッケージをインストールすれば手に入るのか？」という、ファイル名からの逆引き検索を行う特殊部隊です
。

3. 【コマンド解剖】：apt-get の主要サブコマンド
昌敏さんが2回目に引き出した apt-get のサブコマンドのうち、システム運用で必須となる5つのコマンドを解説します。
update（データベースの更新）
 インターネット上の倉庫（リポジトリ）にアクセスし、「どんなソフトウェアの最新版が存在するか」という目次（インデックス）情報だけを更新します。まだソフトウェア本体の更新は行いません
。
upgrade（ソフトウェアの更新）
 update で取得した目次をもとに、現在ラズパイにインストールされているソフトウェアをすべて最新版に更新します
。（※既存のパッケージを削除してまで更新することはしません
）
dist-upgrade（システム全体の強制更新）
 upgrade よりも強力で、古いパッケージの削除や、依存関係の大幅な変更を伴うような、OSのメジャーバージョンアップ相当の深い更新を行う際に使用します
。
install（導入）
 指定したパッケージをインストールします。この時、APTの最大の強みである**「そのソフトを動かすために必要な別のソフト（依存関係）も自動的に計算して一緒にインストールしてくれる」**という高度な処理が裏で行われています
。
remove / purge（撤去）
 remove はパッケージを削除します（設定ファイルは残す）
。purge は設定ファイルも含めて完全にシステムから抹消します
。

2. 【真相解明】：なぜオーディオデバイスが迷子／使用中になったのか？
前回（5月25日）の時点では mpg321 で綺麗に音が出ていたのに、なぜ今日になって突然エラーになったのか。インフラ的視点から、**「本日行ったハードウェアの変更」**が原因であると断言できます。
原因①：USBカメラ（C270）接続に伴う「カード番号のズレ」 前回、イヤホンジャックから音を出すために ~/.asoundrc という設定ファイルに配管の向き先（例：defaults.pcm.card 2 など）を固定で記述しました。 しかし本日、マイクを内蔵している「Webカメラ（C270）」をUSBに接続したことで、Linux OSが「おっ、新しいオーディオデバイス（マイク）が追加されたぞ！」と認識し、サウンドカードの番号（物理的な住所）の割り当て順をずらしてしまった可能性が極めて高いです。結果として、ALSAが指定された番号の出口を見失ってパニックになっています。
原因②：他プロセスによるデバイスの握り込み（占有） 本日テストした guvcview（カメラアプリ）や、バックグラウンドで動いているVLCメディアプレイヤーなどが、オーディオデバイスを「握ったまま（ホールドしたまま）」になっている可能性もあります。
