第2部：【解剖】 コルーチンとサブルーチンの違い
公式ドキュメントの用語集には、アーキテクチャの真理としてこう記されています。

■ サブルーチン（ def func(): ）の限界 通常の関数は、呼び出されると「一番上」から入り、return に到達する「一番下」から出ます

第3部：【解剖】 コルーチンとジェネレータの違い
「途中で一時停止して、また再開できる」と聞くと、先日学ばれた「ジェネレータ（yield）」にそっくりですよね。 実際、公式ドキュメントにも以下のように書かれています。
「ジェネレータ関数はコルーチンにとてもよく似ています。ジェネレータ関数は何度も生成し、1つ以上のエントリポイントを持ち、その実行は一時停止されます。ジェネレータ関数は yield した後で実行の継続を制御できないことが唯一の違いです。」
この2つの決定的な違いは**「何のために一時停止するのか（目的）」**です。
ジェネレータ（yield）の目的： 大量のデータ（100GBのログなど）をメモリに一気に載せないために、**「データを1個ずつ生成・返却するため」**に一時停止します
。主導権は常に呼び出し元（next() を呼ぶ側）にあります
。
コルーチン（await）の目的： I/Oバウンド（通信やファイル読み書きの待ち時間）による無駄をなくし、**「CPUの実行権（主導権）を他の並行タスクに譲るため」**に一時停止します。これを管理するのが「イベントループ（Event Loop）」と呼ばれる高度なスケジューラーです
。
※Python 3.5以降、この2つを文法的に明確に分けるために、コルーチン専用の async def と await というキーワードが導入されました
。

1. 【I2CBUS番号の解剖】：1番以外の配管はあるのか？
結論から言うと、ラズパイには「1番以外」のI2Cバス（配管）も物理的・論理的に存在します。
「1番」がメイン配管である理由： 現在昌敏さんが接続しているラズパイの汎用ピン（GPIO2のSDA、GPIO3のSCL）は、OSのアーキテクチャ上、デフォルトで**「I2Cバス 1番」**として割り当てられています。私たちが普段電子工作で繋ぐメインストリートです。
「0番」の存在（歴史と内部配管）： 実は、2012年に発売された初代Raspberry Pi（Model B Rev1）では、このメインの配管が「0番」でした。 現在のラズパイ（Pi 3, 4, 5など）にも「0番」の配管は存在しますが、これは主に**「カメラモジュール」や「公式ディスプレイ」と通信するための内部的な専用配管（I2Cバス 0番）**としてシステムに予約されています。
その他の番号（上級者向け）： Raspberry Pi 4や5になると、アーキテクチャの進化により、余っている他のGPIOピンを使って「バス2番」「バス3番」といった新しいI2C配管をソフトウェア的に増設（有効化）することも可能です。
2. 【オプションの解剖】：職人の点呼マニュアル
i2cdetect が提示した [-y] [-a] [-q|-r] というオプションは、I2Cバスに繋がっている職人（センサー）たちに対して**「どのように点呼（プローブ）を取るか」を細かく調整するダイヤル**です。
[-y]（Yes：強制実行） 先ほどご自身でも確認された通り、実行時の「本当にやりますか？」という警告プロンプトをスキップして強制実行する、インフラ工事用の効率化オプションです。
[-a]（All：全領域スキャン） 先ほどのログを思い出してください。点呼は 0x08 から始まり、0x77 で終わっていました。実は 0x00〜0x07 と 0x78〜0x7F のアドレスは、I2Cの規格上「特殊なシステム予約席」として普段は点呼から除外されています。-a を付けると、この予約席を含めた**「0x00から0x7Fまでの全アドレス（All）」を強制的にスキャン**します。
[-q] と [-r]（Quick Write と Read：点呼の方式） これは「声をかけられた時の返事のさせ方」を指定する高度なオプションです（どちらか一方しか選べません）。
-q (Quick Write コマンド)：デフォルト（何も付けない時）の挙動です。システムは職人に対して「ちょっと書き込みのテスト信号送るよ」と声をかけ、反応があるかを見ます。
-r (Receive Byte コマンド)：一部のデリケートな職人（センサー）は、-q のテスト信号を受け取るとパニックを起こしてフリーズすることがあります。その場合、この -r を付けて「書き込みではなく、ちょっと読み込み（Read）のテストさせて」という優しい点呼方式に変更します。

1. 【インフラの仕様】：I2Cプロトコルの「予約席」
I2Cという通信規格（プロトコル）には、0番（0x00）から127番（0x7F）までの内線番号（アドレス）が存在しますが、実はこれらすべてを自由に使っていいわけではありません。
絶対不可侵の予約席： I2Cの国際規格により、最初の8個（0x00〜0x07）と最後の8個（0x78〜0x7F）は、「繋がっている全員への一斉送信（General Call）」や「将来の拡張用途」といった特殊機能のための**「システム予約席」**として厳格に確保されています。
i2cdetect の親切心（安全装置）： Linuxの i2cdetect コマンドは、この予約席にむやみにテスト信号（点呼）を送ると、システム全体が誤作動を起こす危険性を知っています。そのため、デフォルトでは安全な一般席である**「0x08 から 0x77 の範囲しか探さない（プローブしない）」**という安全設計になっているのです。
2. 【ハードウェアの事情】：なぜ「04」に座っているのか？
では、なぜ Grove Base Hat のマイコンは、わざわざその不可侵の予約席である「04番（0x04）」に座っているのでしょうか？
真相解明： ソース資料の仕様表にも明確に「I2C Address: 0x04」と記載されている通り
、製造元である Seeed 社が、Hatに搭載したアナログ変換用マイコンのアドレスとして、なぜかこの予約席（0x04）を物理的に割り当ててしまったのです。（※最近のMM32マイコンを搭載した新ロットでは、一般席である 0x08 に修正されていることも資料から読み取れます
）。
3. 【統合結論】：摩擦の正体
つまり、今回の 0x04 がデフォルトで非表示だった理由は以下の通りです。 「Linux OS側は『まさか予約席（0x04）に一般のセンサーが座っているわけがない』と信じて点呼をスキップしたが、中国のハードウェアメーカー（Seeed社）は規格を少し無視して0x04にマイコンを置いていたため、すれ違い（非表示）が起きた」

3. 【アーキテクチャ解説】：smbus2 とは何か？

smbus2 ライブラリの役割： Raspberry Pi の Linux カーネルには、I2C/SMBus 用のデバイスドライバ（カーネル層）が最初から組み込まれています。smbus2 は、Python（ユーザー層）からそのカーネルの配管を安全かつ簡単に叩くための「橋渡し（API）」 です
。 このモジュールのおかげで、私たちは read_i2c_block_data(ADDR, 0x71, 7) のように、「特定のアドレスの、特定の部屋から、7バイトまとめてデータを引っこ抜く」という低レイヤのハードウェア操作を、Python からたった1行で実行できるのです

① shutdown コマンド（伝統的な電源オフ）

1. 【システム解剖】：systemctl に命令できる3つの電源コマンド
最新のLinux（systemd）において、司令塔である systemctl に出せる電源関係の直接命令（サブコマンド）は、以下の3つが存在します
。
① systemctl poweroff（電源切断） システムを安全に停止させた後、物理的な電源を完全にオフにします。これが一般的な「シャットダウン」に相当します
。
② systemctl halt（システム停止） システム（OS）を安全に停止させますが、**「物理的な電源は切らない（通電したままCPUだけ止める）」**という動きをします
。ラズパイではあまり使いませんが、サーバー運用などで使われます。
③ systemctl reboot（再起動） システムを安全に停止させた後、自動的に再起動をかけます
。ターミナルから遠隔操作している時に頻繁に使用します。
