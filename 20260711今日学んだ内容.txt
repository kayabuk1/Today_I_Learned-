結論から申し上げますと、プロンプトの @ の右側に表示されるのは「接続に使った宛先」ではなく、「到着したマシンの本当の名前（ホスト名）」だからです。

どちらの道しるべを使っても、結局到着するのは「Raspberry Pi」という同じ家（自分自身）なので、システムは律儀に「ここは raspberrypi ですよ」と表札を表示しているのです。

よく使われる定義のルールは以下のようになっています

\u：ユーザー名（今回は pi）
\h：ホスト名（今回は raspberrypi）
\w：現在の作業ディレクトリ（今回は ~ ＝ホームディレクトリ）
つまり、システムは接続元の情報などは気にせず、「今ログインしているユーザー名（\u）」と「このマシンのホスト名（\h）」を機械的に組み合わせて画面に出力しているだけなのです。

1. なぜ dnf のサブコマンドは provides （供給）なのか？
Linuxのパッケージ管理システムにおいて、ソフトウェア（パッケージ）は単なるデータの塊ではなく、**「システムに対してコマンドや機能を『提供（Provide）』してくれる箱」**という考え方をします。

🌐 仮想化の全体像（なぜこんなことをするのか？）
大前提として、仮想化とは**「1つの広大な土地（物理サーバー）を、論理的に切り分けて有効活用する技術」**です。その切り分け方に、大きく2つの流派があります。
1. 仮想マシン（VM）方式 ＝ 【独立した一軒家を建てる】
VM（Virtual Machine）は、物理的な土地の上に「完全に独立した一軒家」を何軒も建てる方式です。 この家を建てるための大工さん（管理ソフト）のことを**「ハイパーバイザー」**と呼びます。
【大工さん（ハイパーバイザー）の2つの流派】
① ベアメタル型（建売住宅）： 土地（ハードウェア）の上に、直接大工さん（ハイパーバイザー）が入り、そこに一軒家（ゲストOS）を建てます。無駄がないので、本番の企業サーバーでよく使われます（例：XenやVMware ESXi）。
② ホスト型（居候・間借り）： すでに建っている大家さんの家（WindowsやMacなどのホストOS）の中に、大工さんが入ってきて、その上に無理やり別の一軒家（ゲストOS）を建てます。昌敏さんが今使っている VirtualBox はまさにこれです！
【VM（一軒家）の特徴】
完全独立： 各家ごとに水道管や電気（カーネル＝OSの心臓部）を別々に引いています。
自由度MAX： だから、隣の家がLinuxでも、自分の家はWindowsにすることができます
。
デメリット： 家を丸ごと建てる（OSを1から起動する）ため、重くて起動が遅いです
。別の機械のフリをする「エミュレーション」という重労働をしているためです。
2. コンテナ方式 ＝ 【シェアハウスの個室】
一方のコンテナは、広大な土地に「1つの巨大な家（ホストOS）」だけを建てて、その中をパーティションで区切って住むシェアハウス方式です。
【コンテナ（シェアハウス）の特徴】
カーネル（水道・電気）は共有： VMのように家ごとに水道管を引くことはしません。大家さん（ホストOS）の水道管（カーネル）をシェアハウスの住人全員で共有します
。
個室は完全に隔離： 水道は共有ですが、各住人には「鍵付きの個室（独自のストレージやネットワーク）」が与えられ、隣の部屋は見えません。コンテナ内から見ると「自分だけの家」に見えるという騙し絵のような構造です。
メリット： 水回りを作る必要がないので、入居（起動）が爆速で、超軽量です
。
【最大の制限事項（試験でよく出る引っかけ！）】 水道管（カーネル）を共有しているため、「大家さんの家（ホストOS）の規格に合わない住人（別OS）」は住めません
。
大家さんがLinuxの場合、Windowsコンテナは動かせません（規格が違うから）。
しかし、同じLinux一族（Ubuntu、CentOS、AlmaLinuxなど）であれば、方言が違うだけなので同じシェアハウスに住むことができます
。

ハイパーバイザーの正体（社長）： ベアメタル型のハイパーバイザー（Xenなど）は、ハードウェアの直上で動く本当に極小のプログラムです

2. コンテナの爆速起動と「PID（プロセスID）」の騙し絵

② 「PID（プロセスID）」が違うという騙し絵 コンテナ内のプロセスは、ホストOS上で動くただのプロセスです

2. なぜコマンドの最後に /bin/bash を指定したのか？
問題集では docker run -it centos となっていたのに、なぜ私が /bin/bash を付け足すように指示したのか。ここには、Docker（コンテナ）の非常に美しいロジックが隠されています。
問題集の解説にも、以下のような決定的な記述があります。
「コンテナが最初に生成したプロセス(この例ではbash) を exitするとコンテナは停止します。」
【論理の解剖】
Dockerコンテナの本質： コンテナとはOS全体を起動する仮想マシン（一軒家）とは違い、**「指定された1つのプログラム（プロセス）を動かすための隔離された部屋」**に過ぎません。

コマンドの意味： docker run -it centos:7 /bin/bash とは、「CentOS 7の環境を作って、その中で一番最初の仕事（PID=1）として、Bash秘書を呼び出せ！」という命令になります。
問題集の古い環境（当時の centos:latest）では、イメージの内部に「何も指定されなかったらデフォルトで /bin/bash を起動する」というお節介な設定が含まれていたため、省略しても動きました。 しかし、環境やイメージ（例として問題集にある ubuntu:14.04 /bin/bash
など）によっては、省略すると「何を動かせばいいか分からない」とすぐ終了してしまうことがあります。
だからこそ、明示的に 「コンテナの中でBash秘書を呼び出して、対話（-it）するぞ！」 と指名するために /bin/bash を付け足したのです。

1. -it オプションの正体
資料にも「オプション「-it」(--interactive=true --tty=true)を指定すると擬似端末を割り当て、標準入力からの入力によって対話的にコンテナ内のコマンドを操作できます」と記載されています

これは、-i と -t という2つのショートオプションの合体技です。
-i (--interactive / インタラクティブ) 「対話的」という意味です。通常、コンテナはプログラムを動かしてすぐ終了しますが、このオプションをつけると**「キーボード（標準入力）からの入力を、コンテナの中にずっと繋ぎっぱなしにしてくれ！」**という指示になります。
-t (--tty / ティーティーワイ) 「疑似端末（ターミナル画面）」という意味です。これをつけると、コンテナの中に**「昌敏さんが見やすいような黒い画面（プロンプトなど）の枠組み」**を用意してくれます。これがないと、文字は打てても [root@...]# というプロンプトが表示されず、非常に操作しづらくなります。
【結論】 -it とは、**「コンテナの中に自分のキーボードと画面を直結させて、秘書（Bash）と直接対話するための必須オプション」**なのです。
